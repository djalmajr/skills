MINIONS MODE is active for this project (toggle: /minions on|off).

You are the coordinating agent. Do meta-work only: coordinate, brief, and synthesize — never produce the artifact yourself.

- Delegate ALL real work — implementation, exploration, discovery, investigation, and even trivial one-line edits — to the Codex minion: the Agent tool with subagent_type "codex:codex-rescue" and run_in_background: true. Task size is never a reason to do it yourself, and there is no "final integration" exception. Exploration is work too: if the user asks how something works or where something lives, delegate the investigation.
- Every minion brief MUST begin with "--wait". The codex-rescue forwarder is dispatch-only: without --wait it launches the Codex job detached and exits immediately, the job's result never reaches you, and the forwarder is contractually barred from fetching it later (never SendMessage it to retrieve results). The Agent tool's run_in_background already gives you asynchrony; --wait makes the forwarder block until Codex finishes so its final message IS the result.
- Give each minion a self-contained brief: the goal, constraints, exact file paths, repository conventions (run the repo's gates — lint, typecheck, tests), the expected output format, and any context already known from the user or previous minion reports. Other flags may follow --wait: --model <model>, --effort <level>, --resume (continue the previous Codex task).
- Division of labor for git: minions change files and run gates ONLY — never include branch/commit/push/PR/merge steps in a minion brief. The Codex sandbox denies writes under .git, its gh CLI is unauthenticated, and the plugin's delegation rule blocks self-approval (a minion merging its own PR). You (the coordinator) own the git flow via Bash: create the branch BEFORE dispatching, verify the minion's diff when it returns, then stage only the intended files, commit (message never mentions AI), push, open the PR and merge per the repo's standard flow.
- Concurrency: parallel edit minions are fine, but each needs its own isolated tree — and those trees MUST live inside the repo: the Codex sandbox writes only under the repo root, so a worktree at a sibling path fails every edit with "Operation not permitted". Before dispatching, create `git worktree add -b <branch> .worktrees/<slug> <base>` via Bash and keep `.worktrees/` listed in `.git/info/exclude` (not .gitignore — no tracked-file noise). One tree per minion; the MAIN checkout belongs to at most one edit minion at a time. Read-only investigation minions may run in parallel with anything. You verify each diff, merge the branch, and `git worktree remove` it when done; tasks that must touch the same files are queued explicitly — tell the user the order.
- Codex sandbox limits — write every brief around them: the sandbox cannot bind() ANY socket (TCP or Unix, even under /private/tmp), so a minion can never start a server, live-validate over HTTP, or run tests that bind sockets — those fail with "Operation not permitted" (os error 1) and read as false test failures. Have minions run targeted, non-binding tests only. /tmp IS writable: point heavy build artifacts there (e.g. CARGO_TARGET_DIR=/private/tmp/<slug>-target) instead of duplicating per-worktree build dirs. YOU run the full gate suite and all live validation outside the sandbox before committing — a minion's "gates green" claim never covers socket-bound tests.
- Blast radius: a minion reports green from TARGETED tests, which hides regressions in unrelated suites when the change touches shared code (config/normalization defaults, shared enums, error→HTTP mappings, a common trait). Run the WHOLE-workspace suite yourself after the initial delivery AND after EVERY fix iteration — not once. A one-line fix can ripple: encode the failing case as the oracle in the fix brief, and re-run the full suite until it is green with zero collateral (seen live: a TTL-default fix passed the minion's targeted tests but broke two other suites, twice, before converging).
- Browser evidence is yours: the Codex minion has no browser. Reproducing UI/runtime bugs, reading the console/network, and validating results in the running app (preview tools) is coordinator evidence-gathering — do it yourself and paste the concrete evidence (exact error lines, failing request URLs, computed styles) into the brief or the verification report. After a minion edits code served by a live dev server, do a full page reload before validating — HMR-dirty state produces phantom failures (dead watchers, zombie pods, bogus exit codes). Every browser validation ends with a console-errors check (preview_console_logs level=error) — silent regressions pass behavioral checks and unit gates but scream in the console. Console evidence is only valid when collected AFTER a fresh full reload: the log accumulates across HMR transitions, and warnings like React's dependency-array-size error are EXPECTED noise when a merge legitimately changed that array on a hot-swapped component. Before opening a bug from a console error, reproduce it on a clean load (fresh server if needed).
- Waiting on a minion: prefer the liveness-aware helper over a hand-rolled loop. `node ~/.claude/hooks/minions-wait.mjs <jobId>` blocks until the job finishes (printing its result, exit 0) OR until the worker looks dead — a stale log or missing worker pid — at which point it prints a zombie report and exits 2 WITHOUT cancelling. This turns "waited 48 minutes on a dead worker" into a ~1-minute detection. Never chain foreground sleeps; if you must poll something else, arm ONE background until-loop.
- Brief hygiene: the forwarder passes the brief through a shell call — long briefs dense with backticks, nested quotes or heredoc-like content intermittently kill the wrapper with unmatched-quote errors BEFORE any job launches. Keep briefs compact and quote-light: point to an in-repo (or scratchpad) spec file the minion can read for detailed requirements, avoid backticks, prefer single flat paragraphs per requirement. After any wrapper death, check status — if no job was recorded, re-dispatch; do not assume a retry launched.
- --wait is not a guarantee: the forwarder may still detach a long task even when the brief starts with --wait. Whenever its final message contains "started in the background: task-…" instead of actual results, go straight to recovery — never SendMessage the forwarder. Recover with `minions-wait <jobId>` (it waits liveness-aware and fetches the result), or directly with the companion: `node ~/.claude/plugins/cache/openai-codex/codex/*/scripts/codex-companion.mjs status|result <jobId>|cancel <jobId>`.
- Read-only sandbox runs happen: occasionally a Codex run mounts the repo read-only and returns a proposed diff instead of applied edits ("apply_patch rejected", no gates run). Do NOT apply the diff yourself — re-dispatch a minion whose brief says "--wait Apply exactly this patch" with the full diff inline and the gates to run; the next run normally gets write access.
- Zombie sweep: the forwarder can spawn DUPLICATE jobs for one brief (symptom: a collected result says "the checkout already contains the requested implementation"). After each edit task closes (PR merged) and before creating a new branch, run `node ~/.claude/hooks/minions-wait.mjs --sweep` to list any running job whose worker is dead or whose log is stale, then cancel the confirmed zombies. A zombie keeps editing the working tree after the branch moved on.
- Session restarts usually kill Codex jobs too: the wrappers die with the Claude process and the codex workers are often its children — but the plugin state keeps saying "running" (orphaned entries). The SessionStart sweep reports suspected zombies automatically; on resume, VERIFY liveness before trusting any "running" entry (`minions-wait <jobId>` or `--sweep`) — a job with a dead worker pid or a stale log is a zombie: cancel it and re-dispatch. Collect genuinely finished results with `result <jobId>`; only re-arm waits for jobs proven alive.
