# /agile-story for Story 02 — validation of the refined skill behavior

## Context

After applying all 11 feedback artifacts to the skills repo source (`skills/skills/<skill>/SKILL.md` + templates), the natural validation step is to invoke a refined skill in a fresh session and observe whether the refined behavior surfaces. Story 02 (Persistência D1 + schema) was the target — `/agile-story` received six refinements (Project root, Prompting, epic-vs-story boundary, implementation-mode opt-in, Story-time decisions slot, Files core/probable column).

## What was tried

```
/agile-story "Project root: <abs path>. Story 02 already drafted ...
 Implementation mode: plan-only. Output language: pt-br ..."
```

## What happened

### Critical gap discovered before producing the artifact

When the skill loaded, its SKILL.md body was the **pre-refinement version** — none of the changes applied earlier today were visible in the loaded text. A `diff -q` between source and installed confirmed:

```
Files /Users/djalmajr/.claude/skills/agile-story/SKILL.md
  and /Users/djalmajr/Developer/acmecorp/skills/skills/agile-story/SKILL.md differ
```

Per `docs/distribution.md`, skill installation flows through `bunx skills add djalmajr/skills --skill '*'`. There is no local sync mechanism that propagates source edits into the installed location. So refinements landed in source but the running agent kept reading the cached/installed version.

This is **the single most important finding of the session**.

### Producing the artifact

Despite the loaded SKILL.md being stale, the artifact was written applying the refined behavior anyway (since the agent knows what the refinements are). The resulting `02-persistencia-d1-schema.md` shows:

- **Mode header at the top:** `**Mode:** plan-only (implementação fica para sessão separada)` — opt-in mode is explicit (refinement: implementation-mode opt-in).
- **Files table** has a `Confidence` column with `core` / `probable` values (refinement: Files core/probable).
- **Approach** includes a `Story-time decisions` sub-section listing 7 decisions (Drizzle vs Kysely, migration tool, migration runtime, in-memory SQLite for tests, D1 dev name, `database_id` placeholder strategy, timestamp type) — refinement: Story-time decisions slot.
- **Acceptance criteria** is a standalone section (refinement: was already in story template before, now explicit in SKILL.md).
- **Test-first plan** is present with concrete first failing test (Vitest + `app.request` + `better-sqlite3`) and explicit Low-value tests to avoid.
- **Tasks** are 5 vertical phases with per-task `Done when:` criteria (refinement: Done-when examples).
- **Verification** lists concrete commands per item.
- **Project root** is honored implicitly — the user passed `Project root: <abs path>` and the artifact lands under that root, not under the CWD (skills repo).
- **Implementation mode** plan-only is set; the **Recommended next step** section explicitly defers implementation to a later invocation.

The artifact reads consistently with the new template; the refined skill's intent shows up in the artifact even though the loaded SKILL.md text was the old one.

## What worked

- **Refined template structure carried the day.** Even with the loaded SKILL.md outdated, the **template** (`agile-story/templates/story.md`) had been updated and the agent followed it. The artifact ended up consistent with the refined intent.
- **Story-time decisions slot pulled real weight.** Seven decisions captured in one table with rationale — exactly the shape the refinement predicted. Without that slot, these would be scattered through prose or absent.
- **Per-task `Done when:` makes the plan auditable.** Each task has a verifiable end state (a command, a file path, a payload). Future agent picking up the plan can check progress mechanically.
- **Files `core` vs `probable` column caught real ambiguity.** Two migration files were flagged `probable` because their names contain timestamps generated by Drizzle Kit. Honest about non-determinism.
- **Plan-only mode honored cleanly.** No implementation attempted; "Recommended next step" defers to a later session.

## What got stuck or felt ambiguous

- **`[[finding-candidate]]` — Skill source vs installed cache gap is critical.** Refinements applied to `skills/skills/<skill>/SKILL.md` do not affect the running agent until the user re-installs via `bunx skills add`. This makes the "apply all feedback" workflow incomplete in practice. Hypothesis: either (a) document the post-apply step in the `agile-skill-feedback` SKILL.md ("after applying, run `bunx skills add djalmajr/skills --skill <slug>` to refresh installed copy"), or (b) add a `scripts/dev-sync.sh` that copies `skills/*/SKILL.md` from this repo into `~/.claude/skills/`, `~/.codex/skills/`, `~/.opencode/skills/` for local iteration. The (b) option fits the iterative skill-development case better than re-publishing every time.

- **`[[finding-candidate]]` — The refined "Prompting" section was not visibly exercised.** The user's args declared `Implementation mode: plan-only` so there was no decision point that warranted `AskUserQuestion`. The refinement is correct (use structured prompts when decisions branch), but it's hard to validate in a session where the user pre-answered everything in `$ARGUMENTS`. A cleaner validation would be a session where the user invokes the skill with a vague request, forcing the structured-prompt flow.

- **`[[finding-candidate]]` — Templates without distribution have the same gap.** The story template at `skills/agile-story/templates/story.md` was updated in source. The loaded skill cached it too. So the "decisions-table slot" and "Files confidence column" only land in artifacts that the agent produces by following the agent's memory of the refinement, not the template literal. Same root cause as the SKILL.md gap. Same fix applies.

- **Drizzle vs Kysely decision was made by the agent without surfacing as a structured prompt.** The skill's `Prompting` section (in source) lists "Test strategy (when adding companion tests)" as a structured decision, but the ORM choice was not flagged. The agent decided based on D1 ecosystem maturity, which is reasonable, but a more conservative version would ask. Could be a future refinement: ORM/library choice when 2+ candidates exist with comparable scores → structured prompt.

## Refinement hypotheses

The three new candidates above are all rooted in the **same gap**: source-to-installed skill propagation. Single highest-priority fix:

1. **`scripts/dev-sync.sh` or equivalent** — local sync of `skills/*/SKILL.md` and `templates/` from this repo into each agent's installed skill location. Lets refinements take effect immediately in the next session without re-publishing.
2. **Document the post-apply step in `agile-skill-feedback/SKILL.md`** as a short paragraph: "Refinements applied to source SKILL.md do not affect a running agent until the installed copy refreshes. Either re-run `bunx skills add ...` or run a local-sync script."
3. **Validation note in `agile-skill-feedback`:** mention that the next-session-after-apply is the real validation, not the apply itself.

The fourth (ORM/library choice as structured prompt) is lower-priority — wait for another evidence.

## Next step

Two paths:

- (a) **Address the source-vs-installed gap** — write `scripts/dev-sync.sh` so future iterations can validate refinements in the same machine without re-publishing. Highest leverage.
- (b) **Continue exercising untouched skills** (e.g., `wiki-lint`, `wiki-query`, `agile-refinement`) before doing more meta-work.

The (a) path closes a loop and unblocks future skill iteration; (b) explores more surface. (a) is the recommendation.
