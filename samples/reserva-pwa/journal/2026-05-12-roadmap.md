# /agile-roadmap for the reserva-pwa MVP

## Context

After translating the validation diary and the sample project to English ([findings-promotion session](./2026-05-12-findings-promotion.md) plus the language pass), the next planned step from the intake was `/agile-roadmap`. The intake recommended it because three tracks need sequencing (foundation, customer, restaurant), foundation decisions affect everything else, and total scope exceeds a single epic.

## What was tried

```
/agile-roadmap "Sample project: OpenTable-style restaurant reservation PWA.
 Intake identifies three tracks: foundation, customer, restaurant.
 Two actors, PWA installable + minimal offline, solo dev, no deadline.
 Output in English."
```

## What happened

1. Skill loaded the trajectory-roadmap workflow (six steps from objective → phases → ordering → outcomes → risks → validation).
2. Template at `templates/roadmap.md` was read — small, clean structure.
3. Built a 5-initiative roadmap:
   - **Foundation** — stack, data model, auth, PWA shell, CI.
   - **Restaurant supply (minimum)** — capacity, blackout, reservation window, day view.
   - **Customer demand (minimum)** — discovery, booking, cancellation, "my reservations" offline.
   - **Policy & notifications** — no-show enforcement, PWA push + fallbacks, identity hardening.
   - **Pilot readiness** — LGPD review, observability, onboarding doc.
4. Saved at `<project-root>/planning/mvp-reserva/roadmap.md` (absolute path used explicitly).
5. Ended with the recommended next step: `/agile-epic` for Foundation, plus a flagged open question about whether stack choice runs as a spike or as upfront commitment.

## What worked

- **The intake → roadmap chain is tight.** The intake had already done the work of identifying tracks, constraints, and open questions; the roadmap mostly turned them into a sequence with dependencies. Low friction.
- **The trajectory framing fit better than a date-driven one.** No deadline + solo dev means "Q3 vs Q4" would be fake precision. The roadmap focuses on *outcomes* and *order*, not dates, which the template accommodates.
- **The "Out of commitment" section did real work.** It captured the long list of MVP exclusions from the intake (payment, reviews, loyalty, native, recurring, maps) in a discoverable place. Anything in scope creep should now hit this list first.
- **Parallelism note for solo dev.** The skill's `Collaborative work` section is aimed at 2+ devs; its structure was used here to flag where work *could* split if another dev joined later — useful future-proofing without forcing parallelism today.

## What got stuck or felt ambiguous

- **`[[finding-candidate]]` — Third evidence of the path/CWD pattern.** This skill also uses relative paths in its SKILL.md (`Save at planning/<initiative>/roadmap.md`) without declaring the project root. An absolute path was used based on muscle memory from prior sessions, but a fresh agent on a multi-repo project would face the same ambiguity already flagged for `agile-intake` and `wiki-init`. The proposal at [`../proposals/skills-path-cwd-ambiguity.md`](../proposals/skills-path-cwd-ambiguity.md) already plans to roll this out to `agile-roadmap` and the other agile skills — **this is the trigger** to extend the change now rather than later.

- **`[[finding-candidate]]` — Template's "Period objectives" framing is awkward for trajectory roadmaps.** The template uses `## Period objectives` and `## Out of commitment` — language that fits a quarterly roadmap. For a trajectory roadmap with no calendar period, these labels feel off. The content fits, but it would read better as `## Outcome objectives` or `## Roadmap objectives` for the trajectory case. Hypothesis: the template could fork two variants (trajectory vs. quarterly) or use a more neutral header.

- **`[[finding-candidate]]` — "Period objectives" vs. "Initiatives" overlap.** Outcomes belong in both sections, leading to repetition. The template separates objective (what we want) from initiative (what we do), but there is no shared example showing the tension. Adding one example pair to the template would help.

- **No skill-side check for solo dev vs. team.** The SKILL.md says "When the team has 2+ developers..." but does not call out "if solo, parallelism doesn't apply directly". Hypothesis: add a tiny "Solo dev" note paragraph parallel to the existing collaborative-work block.
