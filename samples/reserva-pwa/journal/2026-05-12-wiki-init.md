# /wiki-init for reserva-pwa

## Context

After `/agile-intake` ([2026-05-12-intake-reserva](./2026-05-12-intake-reserva.md)), initialize the wiki **before** moving on to `/agile-roadmap` — several roadmap decisions depend on domain rules (capacity, no-show, reservation window) that must live in the wiki rather than in code or in the roadmap itself.

## What was tried

```
/wiki-init "Initialize the wiki for the sample project.
 OpenTable-style table reservation PWA. Domain: per-shift capacity,
 no-show, reservation window, blackout dates, cancellation.
 Sample project for validating the agile + wiki skill flow."
```

## What happened

1. **Doctor dry-run** (`bun skills/wiki-init/scripts/wiki-init.ts doctor --project <project-root>`) reported:
   - `wiki_path: wiki (missing)` — expected, new project
   - `qmd_index: reserva-pwa` — inferred from the basename, exactly the default that would have been picked
   - `recommended_topology: local-wiki (suggested)` — correct for a sample project
   - 3 harnesses detected (claude, codex, opencode)
   - 22 planned files (configs + hooks per harness + MCP + QMD wrapper + manifest)
   - QMD 2.1.0 already installed and patched OK
2. **Decisions declared up front** (no-pause mode): local-wiki, index `reserva-pwa`, all three harnesses.
3. **Install --write** — created the 22 files with no errors. Doctor next showed all files `present`.
4. **Wiki content** — `doctor` kept reporting `wiki (missing)` after the install. Why? Because `wiki-init` configures **infrastructure**, not wiki **content**. The minimum structure (`wiki/index.md`, `wiki/CONVENTIONS.md`, `wiki/log.md`, audience subfolders `business/`, `apps/`, `ops/`, `data/`, `sources/`, and `raw/index.md`) had to be created manually. The layout was inferred from reading `wiki-ingest/SKILL.md` (which assumes this convention).
5. **QMD index init** (step 9 of SKILL.md): `<wrapper> collection add <wiki-path> --name reserva-pwa --mask "**/*.md"` → 3 docs indexed. `<wrapper> update` → ok. `<wrapper> embed` → 4 chunks across 3 documents embedded in 2s.
6. **Final doctor** — all green: `wiki_path (exists)`, every file `present`, `installed drift: none`, QMD `status: ok`.
7. Non-managed context was added to the project's `CLAUDE.md` (back-reference to the intake, to the journal, and to the domain to be populated).

## What worked

- **`doctor` is an excellent entry point.** Read-only, exhaustive, and lists planned actions explicitly. High confidence that `--write` will not surprise.
- **The default topology and index name landed correctly.** The script already proposed local-wiki and `reserva-pwa` as index — only confirmation was needed. Defaults were right.
- **Install is idempotent.** Re-running `doctor` after install reported `installed drift: none detected` — clear success signal.
- **The QMD wrapper worked first try.** `collection add` + `update` + `embed` in the SKILL.md-prescribed order, no friction.
- **QMD patches detected.** Doctor reported `patch embed model: ok`, `patch MCP --index: ok`, `patch hyphenated vec/hyde: ok` — patch suite is healthy.

## What got stuck or felt ambiguous

- **`[[finding-candidate]]` — The skill configures infrastructure but does not seed wiki content.** `wiki_path: wiki (missing)` persisted after `--write`. The SKILL.md mentions "If the target project needs an index, initialize QMD..." but does not say "create the minimum index/conventions/log". The minimum structure had to be inferred from `wiki-ingest/SKILL.md`. Hypothesis: `wiki-init` could include an optional "seed wiki" step, or at least an *explicit checklist* listing the expected minimum files and where the template is. **This is the second `[[finding-candidate]]` about a gap between what the skill does and what must be done manually afterwards.**

- **`[[finding-candidate]]` — The skills repo (this meta-repo) does not dogfood its own pattern.** The `CLAUDE.md` in the skills repo references `wiki/index.md`, `wiki/CONVENTIONS.md`, `wiki/log.md` — but the `wiki/` directory **does not exist** here. This is a curious meta-divergence: the wiki-skills repo does not use a wiki. It might be intentional (this is a skills repo, not a domain knowledge repo), but the `CLAUDE.md` references it as if it existed. Hypothesis: either remove the reference (if intentional) or run `/wiki-init` in this repo for consistency (dogfooding). Worth promoting because it affects the pattern's credibility.

- **`[[finding-candidate]]` — Repeat of the intake observation: relative path + CWD.** The QMD wrapper was created under the agent's skill cache (absolute path). The command `<wrapper> collection add <wiki-path>` accepts an absolute path, but the SKILL.md does not emphasize this — an absolute wiki path was passed to avoid ambiguity. Hypothesis: document in the SKILL.md examples that absolute paths are the safe form. This is the **second journal** flagging the same pattern (path/CWD ambiguity) — ready to be a consolidated finding.

- **`[[finding-candidate]]` — `qmd embed` spinner leaves garbage in captured output.** The output includes ANSI escape sequences (`[?25l[?25h██████████████████████████████ 100%`) that pollute logs when captured. Small, but a future session that wants to parse output programmatically will suffer.
