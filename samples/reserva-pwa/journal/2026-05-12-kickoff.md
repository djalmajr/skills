# Kickoff of the skills-validation experiment

## Context

A project idea was requested to exercise both the agile cycle (`/agile-intake` → `/agile-epic` → `/agile-story` → ceremonies) and the knowledge base (`/wiki-init`, `/wiki-ingest`, `/wiki-query`, `/wiki-lint`, `/wiki-policy-check`). Extra constraint: the whole process must be documented so the skills can be refined from evidence.

Three initial picks were floated (personal reading tracker, investment tracker, smart kitchen) and four alternatives were proposed (podcast app, raindrop-like bookmark manager with Chrome extension, restaurant reservation PWA, time-tracking app). The **restaurant reservation PWA** was selected as the primary sample because of dense domain rules (capacity, no-show, blackout dates, reservation window) and the presence of two distinct actors (customer + restaurant owner).

## What was tried

No skill was invoked in this session — it was observation-infrastructure setup. Actions:

1. Inspected the skills repo layout (`docs/`, `skills/`, `wiki/`).
2. Read `agile-intake/SKILL.md` to confirm where the skill writes artifacts (`planning/<initiative>/intake.md`, relative to the CWD).
3. Decided the topology: the **diary** lives in `skills/samples/<project>/` (originally proposed as `docs/skills-validation/`, moved to `samples/` in a later decision the same day), the **sample project** lives outside the skills repo. This separates meta-observations about skills from domain knowledge.

## What happened

Created:

- [`samples/README.md`](../../README.md) — purpose, layout, journal → finding → proposal workflow.
- [`samples/templates/journal-entry.md`](../../templates/journal-entry.md) and [`finding.md`](../../templates/finding.md).
- A draft README in the sample project with a back-reference to the diary.
- `git init` for the sample project (the initial commit was rolled back later because it was not authorized).

## What worked

- `agile-intake`'s SKILL.md is precise about where the artifact lands (`planning/<initiative>/intake.md`), which made the sibling-repo topology trivial to pick.
- The `journal-entry` and `finding` templates have clear slots — fill-out friction is to be tested in the next session.

## What got stuck or felt ambiguous

- **`[[finding-candidate]]`** — on one of the commit attempts, the agent reflexively passed `-c commit.gpgsign=false`, breaking the "never bypass signing/hooks without explicit ask" rule. The harness reminded of the rule and the commit was redone without the bypass. Not a skill defect, but worth tracking whether project skills end up running git ops without that constraint being clear.
- **`[[finding-candidate]]`** — `agile-intake` saves to `planning/<initiative>/intake.md` (relative path). If the skill is invoked with the wrong CWD the artifact lands somewhere wrong. Hypothesis: the expected CWD could be declared more explicitly in the SKILL.md.
